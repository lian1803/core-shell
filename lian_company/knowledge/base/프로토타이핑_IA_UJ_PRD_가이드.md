

## ── 프로토타이핑 & IA/UJ/PRD 가이드 ──

## AI 프로토타이핑 & 바이브코딩 강의 전문 추출

본 문서는 'AI 프로토타이핑 & 바이브코딩' 강의의 전 과정을 기획부터 개발까지 상세히 추출한 내용입니다. 사용자 중심 디자인(UCD) 철학을 기반으로 기획, 프롬프트 작성, AI 도구 활용, 배포, 그리고 개선에 이르는 전체 흐름을 다룹니다.

---

### 1. 핵심 마인드셋: 사용자 중심 디자인 (UCD) & 좋은 디자인 원칙

**사용자 중심 디자인 (UCD) 이란?**
"내가 쓰고 싶은 거" 대신 "사용자가 필요한 거"를 만드는 사고방식.
핵심 질문: "이 기능이 진짜로 사용자의 문제를 해결하는가?"

*   **1단계: 공감(Empathize)**: 사용자를 관찰하고 인터뷰합니다. "언제 불편하세요?"
*   **2단계: 정의(Define)**: 문제를 한 문장으로 정리합니다. "~할 때 ~가 불편하다"
*   **3단계: 해결(Ideate → Test)**: 아이디어 → 프로토타입 → 테스트. 빠르게 만들고 빠르게 검증합니다.

**UCD 적용의 중요성:**
*   `❌ UCD 없이 만들면`: 기능은 많지만 아무도 안 쓰고, 출시 후 전면 재설계 위험.
*   `✅ UCD로 만들면`: 핵심 기능으로 편의성을 높이고, 최소 기능으로 빠르게 검증하여 성공 확률 증가.
*   IA → UJ → PRD 과정이 바로 UCD를 실천하는 방법입니다.

**좋은 디자인이란?**
디자인 = 예쁜 것이 아닙니다. 사용자가 "생각 없이 쓸 수 있는 것"이 좋은 디자인입니다.

*   **1. 명확성**: 보자마자 뭘 해야 하는지 안다 (예: 카카오톡 송금)
*   **2. 일관성**: 같은 동작은 같은 모양 (예: Spotify 재생 버튼)
*   **3. 피드백**: 누르면 반응이 온다 (예: 쿠팡 "장바구니 담김!" 토스트 알림)
*   **4. 단순함**: 없어도 되는 건 빼라 (예: 구글 검색창 하나)

**프롬프트 Before → After (UCD 원칙 반영)**
```
❌ Before: 기능만 나열
투두 앱을 만들어줘. 할 일을 추가하고 삭제하고 완료 표시 할 수 있게 해줘.

✅ After: UCD 원칙 반영
투두 앱을 만들어줘.
- 화면 중앙에 입력창, 엔터로 바로 추가
- 완료 시 체크 애니메이션 + 취소선
- 삭제는 스와이프 (실수 방지: 되돌리기 3초)
- 빈 상태: "오늘 할 일을 추가해보세요" 안내
- 색상: 화이트 배경, 포인트 #4F46E5
- 모바일 우선, 엄지 닿는 영역에 버튼 배치
```
같은 "투두 앱"이라도 사용자 경험을 고려한 프롬프트가 10배 좋은 결과를 만듭니다.

### 2. 프로토타입/데모/MVP/PoC 정확한 구분

다 비슷해 보이지만 목적이 완전히 다릅니다.

*   **프로토타입 (Prototype)**
    *   **목적**: 아이디어를 눈에 보이는 형태로 만들기.
    *   **특징**: 동작하는 화면 시안 (클릭 가능), "이런 느낌의 앱이에요" 확인용.
    *   **언제**: 초기 아이디어 구상 및 사용자 피드백 수집 단계. 오늘 우리가 만드는 것이 여기에 해당.
*   **데모 (Demo)**
    *   **목적**: 고객/투자자에게 "이거 됩니다" 보여주기.
    *   **특징**: 프로토타입을 보여주는 행위, "저희 서비스 한번 보세요!" 발표.
    *   **언제**: 투자 유치, 고객 설득, 발표 등 대외적으로 기능을 시연할 때.
*   **MVP (Minimum Viable Product)**
    *   **목적**: 핵심 기능만 넣어 시장에서 검증.
    *   **특징**: 실제 사용자가 쓸 수 있는 수준, 로그인·결제 등 백엔드 포함. "사람들이 진짜 쓸까?" 테스트.
    *   **언제**: 최소한의 기능으로 시장에 출시하여 실제 사용자 반응을 보고 싶을 때.
*   **PoC (Proof of Concept)**
    *   **목적**: "이 기술/방법이 가능한가?" 증명.
    *   **특징**: 기술적 실현 가능성 확인, 디자인·UX는 신경 안 씀.
    *   **예시**: "AI가 음식 사진 인식 가능?" 테스트.
    *   **언제**: 새로운 기술이나 접근 방식의 실현 가능성을 검증해야 할 때.

**한눈에 흐름 정리**:
PoC → 기술 검증 | 프로토타입 → 형태 확인 | 데모 → 발표·설득 | MVP → 시장 검증
린 스타트업은 "빠르게 만들고 → 빠르게 검증 → 빠르게 개선"하는 반복 과정입니다.

### 3. 웹 개발 기초 이해하기: 프론트엔드, 백엔드, 데이터베이스, API

**프론트엔드 (Front-end)**:
*   사용자(손님)가 직접 눈으로 보고 손으로 만지는 모든 것 (카페 인테리어, 메뉴판, 카운터)
*   **HTML**: 뼈대 · 건물 구조 (내용과 구조 정의)
*   **CSS**: 화장 · 인테리어 (색상, 크기, 위치 꾸미기)
*   **JavaScript**: 동작 · 직원 (클릭, 움직임, 기능 만들기)
*   AI Studio에서 만드는 것은 주로 프론트엔드입니다.

**백엔드 (Back-end)**:
*   손님은 안 보지만 없으면 카페가 안 돌아갑니다 (카페 주방)
*   **서버**: 주방장 (요청을 받아 처리하고 결과를 내보냄, 24시간 대기)
*   **API**: 주문서 (프론트엔드가 백엔드에 요청하는 양식)
*   **데이터베이스 (DB)**: 냉장고/창고 (재료(데이터) 체계적으로 보관)
*   **인증 (Auth)**: 직원 출입증 (누가 어느 공간에 접근할 수 있는지 관리, 로그인·회원가입)
*   Lovable · Firebase Studio가 이 백엔드를 코딩 없이 자동으로 만들어줍니다.

**API 란?**
*   앱과 서버가 대화하는 방법 (배달앱의 주문서)
*   **프론트엔드** → 요청 → **API** → 처리 → **서버** → 저장 → **DB** → 결과 전달 → **프론트엔드**
*   **오늘 수업에서의 API**:
    *   STEP 1~2: API 없이 프론트만 (데이터 저장 ✗)
    *   STEP 4: Supabase API 자동 연결 (저장 ✓)
    *   STEP 5: Firebase API 직접 활용 (저장+인증 ✓)

**프론트엔드 vs 백엔드 비교**
| 구분     | 프론트엔드               | 백엔드                  |
| :------- | :----------------------- | :---------------------- |
| **비유** | 카페 매장 (보이는 공간)  | 카페 주방 (안 보이는 공간) |
| **역할** | 사용자가 보는 화면       | 데이터 처리, 저장       |
| **기술** | HTML, CSS, JavaScript    | Node.js, Python, Java   |
| **도구** | AI Studio                | Supabase, Firebase      |
| **결과물** | 버튼, 카드, 애니메이션   | 로그인, DB, API         |

**풀스택**: 프론트 + 백엔드 모두 만드는 것. Lovable과 Firebase Studio가 한번에 해결합니다.

### 4. IA(정보구조) 작성법

**IA란?**
앱에 어떤 화면이 있고, 각 화면에 어떤 요소가 들어가는지 정리한 구조도입니다.

**작성법 (3단계)**
1.  **메인 화면 나열**: 앱에 필요한 주요 화면(탭)을 적습니다.
2.  **하위 요소 추가**: 각 화면에 들어갈 UI 요소들을 나열합니다.
3.  **네비게이션 결정**: 화면 간 이동 방법 (탭바/햄버거 등)을 결정합니다.

**IA 예시: 프레시키퍼 IA (식재료 관리 앱)**
```
📱 메인 (홈)
├ 식재료 카드 리스트
├ 통계 요약 바 (전체/임박/만료)
└ 빠른 추가 버튼 (+)
📷 스캔
└ 카메라 촬영 / 수동 입력 폼
🍳 레시피
├ 임박 재료 기반 추천
└ 카테고리별 브라우징
⚙️ 설정
├ 알림 설정
└ 프로필
Navigation: 하단 탭바 4개
```

### 5. UJ(User Journey) 작성법

**User Journey란?**
사용자가 서비스를 발견 → 사용 → 재방문하는 전체 흐름. 이 흐름의 각 단계가 나중에 기능 목록(Features)이 됩니다.

**작성법**
1.  **발견/인지**: 사용자가 문제를 느끼는 순간과 감정.
2.  **진입**: 앱 설치, 가입 과정. (필요 기능 명시)
3.  **핵심 경험**: 메인 기능을 처음 사용하는 순간. (필요 기능 명시)
4.  **일상 사용**: 반복적으로 사용하는 패턴. (필요 기능 명시)
5.  **확장/습관화**: 고급 기능 발견, 재방문 유도. (필요 기능 명시)

**UJ 예시: 프레시키퍼 User Journey**
```
1. 발견: "냉장고에 뭐 있었지... 또 버리게 생겼네" → 감정: 짜증, 죄책감
2. 진입: 앱 설치 → Google 로그인 → 필요 기능: 회원가입/로그인 (Auth)
3. 첫 등록: 냉장고 식재료 사진 촬영 → 자동 인식 → 필요 기능: 식재료 등록 (CRUD - Create)
4. 일상 관리: D-day 알림 확인 → 임박 재료 확인 → 필요 기능: 알림, 목록 조회 (Read)
5. 활용: 임박 재료로 레시피 검색 → 필요 기능: 레시피 추천
6. 소비: 사용 완료 체크 / 폐기 처리 → 필요 기능: 수정/삭제 (Update/Delete)
7. 습관화: 주간 절약 리포트 확인 → 필요 기능: 통계
```

### 6. PRD(서비스 기획서) 작성법

**PRD란?**
IA + User Journey를 합쳐서 하나의 서비스 기획서로 정리한 문서. 이것이 곧 프롬프트의 원본이 됩니다.

**PRD 작성 템플릿**
```markdown
[서비스 기획서]
1. 서비스명: _____________
2. 한 줄 설명: _____________
3. 타겟 유저: _____________ (나이, 직업, 상황)
4. 핵심 가치: _____________ (사용자가 얻는 것)
5. 핵심 기능 (User Journey에서 가져오기):
   ① _____________
   ② _____________
   ③ _____________
   ④ _____________
   ⑤ _____________
6. 화면 목록 (IA에서 가져오기):
   - 메인: _____________
   - 2페이지: _____________
   - 3페이지: _____________
   - 네비게이션: _____________
7. 디자인:
   - 스타일: _____________
   - 메인컬러: #_____________
   - 레퍼런스: _____________
8. 기술 스택: _____________
```

**PRD 예시: 에어비앤비 클론 (StayKorea) PRD**
```markdown
[서비스 기획서]
1. 서비스명: StayKorea (스테이코리아)
2. 한 줄 설명: 한국 여행자를 위한 숙소 예약 플랫폼
3. 타겟 유저: 20-30대 국내 여행 좋아하는 직장인
4. 핵심 가치: 취향에 맞는 특색 있는 숙소 발견
5. 핵심 기능 (User Journey에서 추출):
   ① 숙소 검색 (지역, 날짜, 인원, 가격 범위)
   ② 숙소 목록 조회 (카드 리스트, 카테고리 필터)
   ③ 숙소 상세 보기 (이미지 갤러리, 편의시설, 리뷰)
   ④ 위시리스트 저장/삭제 (♥ 토글)
   ⑤ 예약 요청 (날짜/인원 선택, 가격 자동 계산)
   ⑥ 리뷰 작성 (별점 1-5 + 텍스트)
   ⑦ 사용자 인증 (Google Auth)
6. 화면 목록 (IA에서 추출):
   - 메인: 검색 + 카테고리 + 숙소 카드 리스트
   - 검색 결과: 필터 적용된 리스트
   - 숙소 상세: 갤러리 + 정보 + 예약 카드
   - 위시리스트: 저장된 숙소
   - 프로필: 사용자 정보
   - 네비게이션: 하단 탭바 4개
7. 디자인:
   - 스타일: 에어비앤비 참고, 깔끔하고 따뜻한 느낌
   - 메인컬러: #FF385C (에어비앤비 핑크)
   - 배경: #ffffff
   - 레퍼런스: airbnb.com
8. 기술 스택: React + Tailwind + Supabase
```
IA의 화면 → PRD 6번 (화면 목록) | Journey의 기능 → PRD 5번 (핵심 기능)으로 연결됩니다.

### 7. IA→UJ→PRD→바이브코딩 전체 흐름

기획 산출물이 곧 프롬프트 재료가 됩니다.

| 기획 산출물 | 프롬프트에서의 역할 (AI Studio) | 프롬프트에서의 역할 (Lovable / Firebase) |
| :---------- | :----------------------------- | :---------------------------------------- |
| **IA**      | `Sections` (화면 요소)         | `Pages` 항목                              |
| **User Journey** | `Features` 목록 (주요 기능 나열) | `Features (CRUD)` 상세                    |
| **PRD (타겟/가치)** | `Context`                        | `Context` 섹션                            |
| **PRD (디자인)** | `Design` (컬러, 스타일)          | `Design` 상세                             |
| **PRD (기술)** | `Tech Stack` (불필요)            | `Tech Stack` 필수 명시                    |

**전체 흐름 요약:**
IA (화면 구조) → User Journey (사용자 흐름) → PRD (기획서 통합) → 프롬프트 (도구별 변환) → 바이브코딩 (AI 앱 생성) → 사용성 테스트 → 개선 반복

**바이브코딩이란?**
프롬프트(자연어)로 AI에게 코딩을 지시하여 앱을 만드는 방법입니다. 코딩 지식 불필요, 프롬프트 품질 = 결과물 품질, UCD 기획이 핵심입니다.

### 8. AI Studio 프롬프트 작성법

AI Studio는 무료 프론트엔드 중심의 프롬프트 실험실입니다.

**프롬프트 엔지니어링 5원칙**
AI에게 일을 잘 시키는 5가지 핵심 원칙입니다.

1.  **역할(Role) 부여**: "너는 시니어 프론트엔드 개발자야" — AI에게 전문성을 부여합니다.
2.  **맥락(Context) 제공**: 누구를 위한, 어떤 상황인지 배경 정보를 제공합니다.
3.  **구체적 지시(Task)**: 기능, 섹션, 동작을 구체적으로 명시합니다.
4.  **디자인 시스템(Design)**: 컬러, 폰트, 레퍼런스 사이트를 지정합니다.
5.  **제약 조건(Constraints)**: 금지사항과 필수사항을 명확히 설정합니다.

**System Instructions (AI Studio만의 무기)**
AI에게 역할과 규칙을 한 번만 설정하면 대화 내내 유지됩니다.
*   **설정 위치**: AI Studio → 화면 오른쪽 패널 → Advanced settings → System Instructions 입력.
*   **예시**:
    ```
    너는 시니어 프론트엔드 개발자야. 다음 규칙을 항상 따라:
    1. 모든 코드는 한국어 주석 포함
    2. 반응형 디자인 필수 (모바일 우선)
    3. 색상은 #0f172a (메인) + #10b981 (포인트)만 사용
    4. Pretendard 폰트 사용
    5. 시맨틱 HTML 구조 사용
    ```

**AI Studio 프롬프트 예시 (V1: 초보 → V3: 고도화)**
```
V1: 초보 ★☆☆☆☆
자기소개 페이지 만들어줘

V2: 구조화 ★★★☆☆
너는 웹 디자이너야. 28세 직장인 김민수의 자기소개 페이지를 만들어줘.
취미: 요리, 자전거
스타일: 미니멀, 다크 테마
한국어 UI로 만들어줘.

V3: 고도화 ★★★★★
Role: 시니어 프론트엔드 개발자
Task: 포트폴리오 자기소개 SPA 제작
Context: 28세 UX 디자이너 김민수
Design:
  - 참고: Apple.com 미니멀 스타일
  - Color: #0f172a + #10b981
  - Font: Pretendard
  - 반응형 (모바일 최적화)

Tech: React + Tailwind CSS
```
Role → Context → Design → Tech 순서로 정보를 추가할수록 결과물이 정교해집니다.

**수정 요청 잘 하는 법**
*   **나쁜 수정 요청**: "더 예쁘게 해줘", "색깔 좀 바꿔줘" → 모호해서 AI가 엉뚱하게 수정.
*   **좋은 수정 요청**: "헤더의 배경색을 #1e293b로 바꾸고, 카드 간격을 24px로 늘려줘", "하단 탭바에 '알림' 탭을 추가하고 아이콘은 종 모양으로 해줘" → 위치 + 속성 + 값이 명확.
*   **Tip 1**: 한 번에 하나씩 수정 요청 — 여러 개를 한꺼번에 하면 AI가 혼란.
*   **Tip 2**: 위치 + 속성 + 값을 명시 — "어디의 무엇을 어떻게".
*   **Tip 3**: 변경 전/후를 비교 확인 — 의도대로 바뀌었는지 검증.

**AI Studio 실습 – 이것만 기억하세요! (프롬프트 패턴)**
*   `✅ 잘 되는 프롬프트 패턴`:
    *   기능을 번호 매겨서 나열하기.
    *   디자인에 구체적 색상 코드 포함 (예: `#3b82f6`).
    *   "모바일 최적화" 명시하기.
    *   "한국어 UI로 만들어줘" 마지막에 추가.
    *   "샘플 데이터 N개 포함" 요청하기.
*   `🔄 안 되면 이렇게`:
    *   새로고침 후 다시 시도.
    *   프롬프트를 더 짧게 줄여보기.
    *   기능을 3개로 줄이고 나중에 추가 요청.
    *   에러 메시지를 그대로 복사해서 채팅에 붙여넣기.

### 9. Lovable, Firebase Studio, Google Stitch 활용법

세 가지 AI 도구는 각각의 장점과 활용 방식이 있습니다.

#### 9.1. Lovable (풀스택 CRUD 앱)

*   **특징**: 유료(무료 체험), 풀스택 (프론트+백엔드), Supabase DB 자동 연결, CRUD 기능 구현, 자동 배포.
*   **Lovable 시작하기**:
    1.  lovable.dev 접속, 구글 계정으로 로그인.
    2.  "New Project" 클릭.
    3.  프롬프트 입력 (다음 슬라이드의 프롬프트를 붙여넣기).
    4.  2-3분 기다리기 → 전체 앱 생성!
*   **Lovable 실전 워크플로우 7단계**:
    1.  **아이디어 선정**: AI에게 추천 요청.
    2.  **기획**: Lovable이 묻는 질문에 답변.
    3.  **첫 번째 프롬프트 → 앱 생성**: 가장 오래 걸림.
    4.  **빠진 기능 확인 + 하나씩 추가**: MVP 기능 중 빠진 것 확인, 하나씩 추가 요청.
    5.  **에러 발생 → 스크린샷으로 디버깅**: 스크린샷 찍어 Lovable 채팅에 붙여넣기.
    6.  **디자인 업그레이드 + 모바일 최적화**: 디자인 텀 사용, 모바일 뷰 확인.
    7.  **보안 스캔 + Publish 배포**: Security 탭에서 스캔, Publish로 배포.
*   **핵심 팁 5가지 Lovable 잘 쓰는 비법**:
    1.  **Plan 모드 활용하기**: AI가 계획 후 코딩하여 더 좋은 결과물.
    2.  **한 번에 하나씩 요청**: 여러 기능을 한꺼번에 요청하면 AI 혼란.
    3.  **에러 = 스크린샷 디버깅**: 에러 화면 스크린샷 첨부하여 AI에게 수정 요청.
    4.  **모바일 뷰 꼭 확인**: 오른쪽 위 디바이스 전환 버튼으로 항상 모바일 뷰 확인.
    5.  **Visual Edit 모드**: 채팅 대신 특정 영역만 클릭하여 수정 요청 가능 (작은 디자인 수정).
*   **디자인 업그레이드 (디자인 용어 활용)**:
    *   `Liquid Glass`, `Minimal Elegance`, `Neumorphism`, `Glassmorphism`, `Editorial`, `Monochrome Luxury` 등 디자인 텀을 프롬프트에 사용하면 AI가 정확하게 이해합니다.
    *   **프롬프트 예시**: "전체적인 디자인을 Minimal Elegance 스타일로 바꿔줘. 그레이디언트를 줄이고 고급스럽고 깔끔한 느낌으로."
*   **Lovable에서 앱 배포하기**: Publish 버튼 2번 클릭으로 전 세계 접속 가능한 URL 생성 (`xxx.lovable.app`).
    *   **Preview**: 개발 중 미리보기 (Lovable 내에서만 확인, 일부 기능 제한).
    *   **Publish**: 실제 배포 (고유 URL, DB 연동 정상 작동).
*   **Lovable 수정 요청 팁 (4가지 패턴)**:
    *   **UI 수정**: "카드 디자인을 더 둥글게 (border-radius: 16px)로 바꾸고, 그림자를 부드럽게 해줘"
    *   **기능 추가**: "식재료 추가할 때 카테고리를 선택할 수 있는 드롭다운 메뉴를 넣어줘. 카테고리: 채소, 과일, 유제품, 육류, 기타"
    *   **버그 수정**: "삭제 버튼을 누르면 바로 삭제되지 않고, '정말 삭제하시겠습니까?' 확인 모달이 먼저 뜨도록 해줘"
    *   **데이터 변경**: "카테고리 목록에 '냉동식품'과 '음료'를 추가해줘"
    *   **핵심**: 한 번에 하나씩!

**CRUD 개념 및 Lovable/Firebase 프롬프트 작성법**
*   **CRUD란?**: Create(생성), Read(조회), Update(수정), Delete(삭제). 모든 앱의 기본 4가지 동작.
*   **CRUD 프롬프트 공식**:
    ```
    [앱 이름]을 만들어줘.
    Features (CRUD):
    - Create: [무엇]을 [어떻게] 추가
    - Read: [무엇]을 [어떤 형태]로 보기
    - Update: [무엇]을 [어떻게] 수정
    - Delete: [무엇]을 삭제 (+ 확인 팝업)
    ```
    앱을 기획할 때 "이 앱의 CRUD가 뭐지?"를 먼저 정리하면 프롬프트가 훨씬 명확해집니다.

**Supabase 연동 (Lovable 백엔드)**
*   **Supabase란?**: 백엔드를 코딩 없이 만들어주는 도구. Database (PostgreSQL), Authentication (로그인), Storage (파일 저장), Realtime (실시간 업데이트) 기능을 제공.
*   Lovable이 Supabase를 자동으로 연결해줍니다.
*   **Supabase 기본 사용법 (Lovable 자동 연동 후 구조 이해용)**:
    1.  supabase.com 로그인 → New Project 생성 (이름, 비밀번호, 리전).
    2.  Table Editor에서 테이블 생성 및 컬럼 추가 (id, name, expiry_date 등).
    3.  Insert Row로 데이터 추가.
    4.  Authentication → Providers → Google 활성화.

#### 9.2. Firebase Studio (Google의 올인원 풀스택)

*   **특징**: 무료, 프론트+백엔드, Firestore DB + Auth 내장, Next.js 기반, 실시간 미리보기, 자동 배포.
*   **Firebase란?**: Google이 만든 올인원 백엔드 플랫폼. Firestore (NoSQL DB), Authentication (구글/이메일 로그인), Hosting (앱 공개), Cloud Functions (서버 코드 실행) 제공.
*   **Firebase Studio 시작하기**:
    1.  firebase.studio 접속, Google 계정으로 로그인.
    2.  "Create an app with AI" 선택.
    3.  프롬프트 입력 (Next.js + Firebase, Google Auth 명시 필수).
    4.  Blueprint 확인 → Prototyping → Code → Preview.
    5.  배포: App Hosting → "Publish".
*   **Firebase Studio 프롬프트 작성법**:
    *   Lovable보다 간결해도 되지만, **기술 스택 (Next.js + Firebase)**과 **인증 (Google Auth)**을 반드시 명시해야 합니다.
    *   **프롬프트 예시 (프레시키퍼)**:
        ```
        프레시키퍼 - 냉장고 식재료 관리 웹앱을 만들어줘.
        주요 기능:
        1. 사용자 로그인 (Google Auth)
        2. 식재료 등록 (이름, 카테고리, 유통기한)
        3. 대시보드에서 D-day 색상 표시
        4. 유통기한 임박 알림
        5. 카테고리별 필터링
        6. 소비완료/폐기 처리
        기술 스택: Next.js + Firebase (Firestore + Auth)
        디자인: 깔끔한 카드 UI, 모바일 반응형
        메인 컬러: #10b981
        한국어 UI, 샘플 데이터 포함
        ```
*   **Firebase Console에서 데이터 확인하기**:
    *   console.firebase.google.com 접속 → 프로젝트 선택.
    *   Firestore Database: 컬렉션/문서 구조 확인 및 데이터 관리.
    *   Authentication: 로그인한 사용자 목록 확인 및 로그인 방법 설정.
    *   Hosting: 배포된 앱 URL 및 배포 기록 확인.

**Supabase vs Firebase 최종 비교**
| 항목         | Supabase           | Firebase              |
| :----------- | :----------------- | :-------------------- |
| **DB 타입**  | SQL (PostgreSQL)   | NoSQL (Firestore)     |
| **데이터 구조** | 테이블 (엑셀 표)   | 컬렉션/문서 (폴더/파일) |
| **인증**     | Supabase Auth      | Firebase Auth         |
| **호스팅**   | 별도 필요          | Firebase Hosting 내장 |
| **연동 도구** | Lovable            | Firebase Studio       |
| **추천 상황** | 정형 데이터, CRUD 앱 | 유연한 데이터, 실시간 앱 |

#### 9.3. Google Stitch (UI 디자인 프로토타입)

*   **특징**: 무료, 텍스트 프롬프트로 UI 디자인 (HTML/CSS) 생성, Variant로 여러 디자인안 비교, 디자인 감각 없어도 프로 수준 UI.
*   **Stitch 사용법 상세 가이드**:
    1.  stitch.withgoogle.com 접속, Google 계정으로 로그인.
    2.  "Design with AI" 클릭.
    3.  디바이스 타입 (Web/Mobile/Tablet) 및 모델 (Gemini 3 Flash/Pro) 선택.
    4.  프롬프트 입력 (영어 프롬프트 권장, 한국어 UI는 명시).
    5.  디자인 확인 → 채팅으로 수정 요청.
    6.  Variant 생성 (여러 디자인안 비교).
    7.  Export (Code 복사 / AI Studio로 보내기 / 스크린샷 저장).

**Stitch 콜라보 워크플로우 (디자인은 Stitch, 구현은 AI 도구)**
1.  **Stitch + AI Studio**: 가장 쉽고 빠른 방법.
    *   **연동 방식**: Stitch Export 버튼으로 직접 연동.
    *   **흐름**: Stitch (디자인) → Export ("AI Studio" 선택) → AI Studio (HTML 코드 + 디자인 이미지 전달, 기능 프롬프트 추가) → Build → Deploy.
2.  **Stitch + Lovable**: CRUD 앱 구현.
    *   **연동 방식**: Stitch 디자인 스크린샷 캡처 후 Lovable 프롬프트에 이미지 첨부.
    *   **흐름**: Stitch (디자인) → 스크린샷 캡처 → Lovable (새 프로젝트, 이미지 첨부 + 기능 프롬프트 작성, "이 디자인 참고" 명시) → Supabase 연동 (선택).
3.  **Stitch + Firebase Studio**: 풀스택 앱 (Auth + DB + 배포).
    *   **연동 방식**: Stitch에서 HTML 코드 복사 후 Firebase Studio 프롬프트에 포함.
    *   **흐름**: Stitch (디자인) → Export ("Copy Code" 클릭) → Firebase Studio (프롬프트 + HTML 코드 붙여넣기, 기술 스택 명시) → Build → Publish.

**Stitch 콜라보 비교 정리**
| 항목             | Stitch + AI Studio | Stitch + Lovable          | Stitch + Firebase Studio |
| :--------------- | :----------------- | :------------------------ | :----------------------- |
| **연동 방식**    | Export 버튼 (직접) | 스크린샷 + 프롬프트       | HTML 코드 복사           |
| **백엔드**       | 없음 (프론트만)    | Supabase 연동 가능        | Firebase (Auth+DB+배포)  |
| **난이도**       | ★★☆☆☆            | ★★★☆☆                   | ★★★★☆                  |
| **추천 상황**    | 간단한 도구/대시보드 | 데이터 저장 필요한 앱     | 로그인 + 실시간 앱       |

### 10. 바이브코딩 실전 팁

앞서 언급된 모든 팁을 포함하여 실전에서 바이브코딩을 효과적으로 활용하기 위한 팁입니다.

*   **수정 요청은 한 번에 하나**: 여러 기능을 동시에 요청하면 AI가 혼란스러워하거나 빠뜨립니다. 하나의 요청에는 하나의 기능 변경만!
*   **구체적으로 요청**: "더 예쁘게" 보다는 "헤더의 배경색을 #1e293b로 바꾸고, 카드 간격을 24px로 늘려줘"와 같이 `위치 + 속성 + 값`을 명확히 명시합니다.
*   **Plan 모드 활용 (Lovable)**: AI가 먼저 계획을 세운 후 코딩하게 하여 더 좋은 결과물을 얻습니다.
*   **에러 발생 시 스크린샷 첨부**: 에러가 발생하면 당황하지 않고 해당 화면을 스크린샷으로 찍어 AI 채팅에 첨부하고 수정 요청합니다.
*   **모바일 뷰/반응형 디자인 확인**: 대부분의 사용자는 모바일이므로, 항상 모바일 뷰를 확인하고 "모바일 최적화 해줘"라고 요청합니다.
*   **Visual Edit 모드 활용 (Lovable)**: 작은 디자인 수정은 채팅 대신 Visual Edit 모드를 사용하여 화면에서 직접 클릭하여 요청합니다.
*   **디자인 용어(텀) 활용**: `Minimal Elegance`, `Glassmorphism` 등 디자인 스타일 용어를 사용하면 AI가 정확하게 이해하고 반영합니다.
*   **CRUD 공식 활용**: 앱의 핵심 기능을 `Create`, `Read`, `Update`, `Delete`로 정리하여 프롬프트에 명시하면 AI가 필요한 기능을 빠짐없이 구현합니다.
*   **기획 선행**: 프롬프트 입력 전 30초라도 IA, UJ, PRD를 간략하게 정리하는 습관을 들입니다. "기획 없는 바이브코딩 = 예쁜 쓰레기"
*   **시스템 인스트럭션 활용 (AI Studio)**: AI Studio에서는 `System Instructions`를 통해 역할과 규칙을 미리 설정하여 일관된 결과물을 얻습니다.
*   **GitHub 연동**: Lovable 등으로 생성한 코드는 GitHub에 연동하여 백업하고, Antigravity 같은 전문 에디터에서 직접 수정하며 Vercel 등으로 자동 배포할 수 있습니다.
*   **Just Ship It 마인드셋**: 완벽하게 만들려 하기보다 "빨리 출시하고 반응을 보고 개선"하는 린 스타트업 방식을 따릅니다. 100개 중 1개만 성공해도 충분합니다.

### 11. Git/GitHub 기초 & Antigravity 활용법

#### 11.1. Git/GitHub 기초

**Git 이란?**: 코드의 "타임머신" — 언제든 과거로 돌아갈 수 있는 코드 변경 기록 관리 도구.

*   **핵심 명령어**:
    *   `commit` (커밋) = "박스에 담기": 변경한 코드를 하나의 박스에 묶어서 설명 메모를 붙이는 것.
        ```bash
        git commit -m "로그인 기능 추가"
        ```
    *   `push` (푸시) = "택배 보내기": 내 컴퓨터의 코드 박스를 GitHub(인터넷 창고)에 보내는 것.
        ```bash
        git push
        ```
    *   `pull` (풀) = "택배 받기": GitHub에 있는 최신 코드를 내 컴퓨터로 가져오는 것.
        ```bash
        git pull
        ```
    *   `clone` (클론) = "복사해오기": GitHub에 있는 프로젝트를 처음으로 내 컴퓨터에 통째로 복사하는 것.
        ```bash
        git clone https://github.com/유저이름/프로젝트이름
        ```
*   **GitHub 란?**: 코드를 저장하는 인터넷 창고 — 전 세계 개발자가 사용합니다.
    *   **Repository (저장소)**: 프로젝트 하나 = 저장소 하나. 코드 + 변경 기록 보관.
    *   **Branch (가지)**: 원본을 건드리지 않고 새 기능을 따로 개발. 완성되면 합침(merge).
    *   **배포 연동**: GitHub에 push하면 Vercel, Netlify 등이 자동으로 웹사이트 업데이트.
*   **GitHub 가입**: github.com에서 무료 계정으로 가입.

#### 11.2. Antigravity (Google AI 코드 에디터)

*   **Antigravity란?**: 구글이 만든 AI 코드 에디터 — 대화로 코드를 직접 수정할 수 있습니다.
    *   Gemini 3 Pro / Claude / GPT 모델 선택 가능.
    *   Planning 모드 (계획 후 실행) / Fast 모드 (바로 실행).
    *   무료 사용 가능 (~2시간/일).
*   **Antigravity 설치**:
    1.  antigravity.google 접속 → Download.
    2.  설치 파일 실행 → Google 계정으로 로그인.
    3.  Open Folder로 프로젝트 열기 (GitHub 클론 폴더).
*   **Lovable vs Antigravity**:
    *   **Lovable**: 대화로 앱 생성, 난이도 낮음, 자동 생성 코드, 자동 배포, Supabase 백엔드.
    *   **Antigravity**: 코드를 직접 수정, 난이도 높음, 직접 편집 코드, GitHub → Vercel 배포, Firebase 등 백엔드.
    *   **시너지**: Lovable로 80% 만들고 → Antigravity로 나머지 20%를 세밀하게 수정하여 최고의 결과물.
*   **Lovable → GitHub → Antigravity 워크플로우**:
    1.  Lovable에서 앱 만들기 (프롬프트로 프로토타입 생성).
    2.  GitHub 연동 → 코드 Push (Lovable 설정에서 GitHub 연결).
    3.  Antigravity에서 GitHub 폴더 열기 (`git clone` 후 Open Folder).
    4.  AI에게 수정 지시 ("버튼 색상을 파란색으로 바꿔줘").
    5.  `git push` → Vercel 자동 배포 (수정 사항 웹사이트에 반영).
*   **Antigravity에서 자주 하는 수정**:
    *   **디자인 커스터마이징**: "전체 테마를 다크모드로 바꿔줘. 배경색은 #1a1a2e, 텍스트는 #e5e5e5"
    *   **새 기능 추가**: "메인 페이지에 검색 기능을 추가해줘. 검색어를 입력하면 리스트를 필터링"
    *   **외부 API 연동**: "OpenWeather API를 연동해서 메인 화면에 현재 날씨를 표시해줘"

### 13. 사용성 테스트 (Usability Test)

내 의도대로 사용자가 움직이는지 확인하는 최종 관문입니다.

*   **Think Aloud (소리내어 생각하기)**: 사용자에게 "사용하면서 드는 생각을 계속 입밖으로 말해주세요"라고 요청합니다.
*   **테스트 시나리오 작성법**: 답을 알려주지 말고 목표만 줍니다.
    *   **나쁜 예**: "우측 상단 메뉴를 눌러서 검색하세요"
    *   **좋은 예**: "원하는 식재료를 추가해보세요"
*   **Nielsen의 법칙**: 5명만 테스트하면 문제의 85%를 발견할 수 있습니다.
*   **문제 심각도 분류**:
    *   **Critical**: 과업을 완수하지 못함 (버그, 길 막힘)
    *   **Major**: 헤매다가 겨우 성공함
    *   **Minor**: 불편하지만 쓸만함
    *   **Cosmetic**: 디자인/텍스트 문제

### 14. 과제 안내 및 실습 치트시트

**과제: 나만의 앱 완성하기 (3단계)**
모든 단계에 기획(IA → UJ → PRD) → 프롬프트 과정이 포함됩니다. 블로그에 기록하여 제출합니다.

*   **🌱 새싹반 (필수)**: 게임 2개 만들기
    *   기획서 (IA + User Journey)
    *   AI Studio 게임 2개 (주사위, 운세, 스톱워치, 팁계산기 등)
    *   제출물: 기획서 + 프롬프트 + 스크린샷 3장+
*   **🔥 열정반 (도전)**: 새싹 + PRD + 수정
    *   기획서 (IA → UJ → 간단 PRD: 서비스명, 타겟, 핵심기능 3개)
    *   AI Studio 게임 2개 (카드짝맞추기, 스네이크, 슬롯머신, 그림판 등)
    *   수정 요청 2회 (디자인/기능 수정 → 전/후 비교 기록)
    *   제출물: 기획서 + 프롬프트 + 수정 비교 + 스크린샷
*   **🚀 심화반 (고수)**: 열정 + 고도화 V1→V3
    *   기획서 (IA → UJ → PRD: 페르소나, 핵심기능 5개+, 디자인 방향)
    *   게임/앱 2개 (두뇌트레이닝, RPG, 피아노, 나라퀴즈 등) + System Instructions 활용
    *   V1 → V2 → V3 고도화: 같은 앱 3번 반복 개선 + 과정 기록
    *   제출물: 기획서 + 프롬프트 + V1~V3 비교 + 스크린샷
