# CPO 분석 -- 네이버 플레이스 자동 진단 + 영업 제안서 시스템

## 제품 전략
소상공인 사장님이 "내 가게가 네이버에서 이렇게 손해보고 있었어?"라고 깨닫게 만드는 자동 진단 시스템.
진단 과정 자체가 계약의 이유가 되도록 설계한다.

---

## 타겟 + Pain

- **타겟 (1차 사용자)**: 리안 본인 및 리안 팀의 영업 담당자
- **타겟 (최종 고객)**: 네이버 플레이스에 등록된 소상공인 사장님 (미용실, 음식점, 학원 등 지역 기반 업종)
- **Pain (리안)**: 영업 미팅 전 제안서 만드는 데 시간이 너무 걸림. 업체당 30분~1시간 → 하루 처리량 한계
- **Pain (사장님)**: 네이버 플레이스를 등록만 해놓고 방치 중. 경쟁사가 어떻게 하고 있는지 모름. "잘 되고 있겠지"라는 막연한 감
- **아하 모먼트**: 사장님이 PPT를 보는 순간 "경쟁사는 리뷰가 320개인데 나는 12개네..." → 격차를 숫자로 인지하는 그 순간

## "계약 욕구를 만드는" PPT 심리 설계 -- 핵심 흐름

### 현재 PPT 구조 (8슬라이드)의 문제
현재는 정보 나열형이다. "사진 몇 장, 리뷰 몇 개, 순위 몇 위" → 사장님 반응: "그래서요?"
**정보를 보여주는 게 아니라, 감정을 움직여야 한다.**

### 제안하는 PPT 심리 흐름 (5단계 감정 곡선)

```
[1] 인정       → "사장님 가게 이런 점은 잘 하고 계시네요"
[2] 충격       → "그런데 이 부분은 심각합니다"
[3] 비교 공포  → "경쟁사는 이렇게 하고 있습니다"
[4] 구체적 손실 → "이것 때문에 매달 이만큼 잃고 있습니다"
[5] 해결 제시  → "저희가 이걸 이렇게 바꿔드립니다"
```

### 슬라이드별 설계

**슬라이드 1: 표지 + 신뢰 확보**
- "OOO 네이버 플레이스 무료 진단 보고서"
- 리안 회사 로고 + 담당자 이름 + 연락처
- 목적: 전문 업체가 만든 공식 보고서라는 인상

**슬라이드 2: 현재 상태 요약 (인정 단계)**
- 업체 기본 정보 (상호, 업종, 주소)
- 잘 되어 있는 항목 먼저 나열 (사진 O, 영업시간 O 등)
- 목적: "우리 가게를 제대로 봤구나" → 신뢰 형성
- 심리: 사람은 자기를 인정해주는 상대의 말을 듣는다

**슬라이드 3: 종합 점수 (충격 단계)**
- 100점 만점 절대 점수 크게 표시: "38점"
- 22개 항목 중 미달 항목 빨간색 하이라이트
- 카테고리별 점수: 기본 세팅 60% / 콘텐츠 20% / 고객 반응 30% / 기능 연동 10%
- 목적: 숫자 하나로 "내 가게가 이 정도밖에 안 되나?" 각인
- 심리: 구체적인 점수는 모호한 설명보다 10배 강하다

**슬라이드 4: 경쟁사 비교 (비교 공포 단계) -- 가장 중요**
- 동일 키워드 상위 3~5개 업체와 나란히 비교 표
- 항목: 리뷰 수 / 사진 수 / 블로그 노출 / 저장 수 / 새소식 업데이트 여부
- "상위 평균 대비 34% 수준"이라는 문장
- 핵심 격차 3개를 빨간 박스로 강조
- 목적: "나만 이렇게 못하고 있었어?" → 불안감 생성
- 심리: 사람은 절대 수치보다 남과의 비교에서 더 큰 압박을 느낀다

**슬라이드 5: 손실 환산 (구체적 손실 단계)**
- 키워드 월간 검색량 제시 → "이 키워드로 월 8,200명이 검색합니다"
- 현재 순위와 클릭률 연결 → "42위는 클릭률 0.3% = 월 25명만 유입"
- 상위 3위 클릭률과 비교 → "3위면 월 820명 유입 가능"
- "매달 약 795명의 잠재 고객을 놓치고 있습니다"
- 목적: 추상적 문제를 돈/고객수로 환산하여 긴급성 부여
- 심리: 손실 회피 본능 -- 사람은 얻는 것보다 잃는 것에 2배 민감

**슬라이드 6: 키워드 전략 (해결의 실마리)**
- 메인 키워드 + 확장 키워드 7개 리스트
- 각 키워드별 월간 검색량
- "이 키워드들을 잡으면 월 OO명 추가 유입 가능"
- 목적: 구체적인 기회를 보여주며 "방법이 있구나" 인식

**슬라이드 7: 솔루션 요약 (해결 제시)**
- 패키지 3종 간략 소개 (시선/집중/주목)
- 각 패키지가 해결하는 항목 매핑
- "시선 패키지만으로도 현재 38점 → 70점 이상 가능"
- 목적: 해결책이 존재하고, 가격 대비 효과가 있음을 인지

**슬라이드 8: CTA (행동 유도)**
- "무료 상세 진단 예약" 또는 "1주일 체험 시작"
- 담당자 연락처, QR코드
- 긴급성: "이 보고서 기준 데이터는 OO월 OO일 기준입니다. 경쟁사는 계속 변합니다."

## MVP 범위 (Must Have) -- Phase 1

아래 기능만 있으면 내일이라도 영업 나갈 수 있다.

### 1. 기본 크롤링 안정화 (현재 코드 수정)
- 현재 process.py가 이미 대부분 동작하지만 PPT 출력물에 URL 인코딩 깨짐, 데이터 0 표시 등 문제 있음
- 크롤링 실패 시 fallback 처리 (빈 값 대신 "수집 불가" 표시)
- API 키 하드코딩 제거 → .env 파일로 이동 (현재 코드에 네이버 API 키가 소스에 박혀 있음)

### 2. 22개 진단 항목 → 100점 절대 점수 계산 로직
- 5개 카테고리 (기본 세팅 / 콘텐츠 품질 / 고객 반응 / 기능 연동 / 외부 채널)
- 항목별 가중치 적용, 100점 만점으로 환산
- 이것만 있어도 "38점"이라는 숫자가 사장님에게 충격을 줌

### 3. 경쟁사 비교 크롤링
- 동일 키워드로 네이버 플레이스 검색 → 상위 5개 업체 place_id 추출
- 각각 동일하게 크롤링 → 비교 테이블 생성
- "상위 평균 대비 OO%" 계산
- 이것이 계약 전환율 1위 요소 (리서치 결과 기반)

### 4. PPT 템플릿 재설계 (위 심리 흐름 반영)
- 5단계 감정 곡선에 맞춘 슬라이드 순서
- 경쟁사 비교 표 자동 삽입
- 점수 시각화 (게이지 또는 큰 숫자)

### 5. 엑셀 배치 처리
- 현재는 첫 번째 행만 처리 → 전체 행 순회로 변경
- 업체당 PPT 1개씩 자동 생성
- 100개 업체 엑셀 넣으면 100개 PPT 나오는 구조

## MVP 제외 (나중에) -- Phase 2 이후

- **웹 대시보드 (프론트엔드)**: PPT로 충분. 웹은 계약 후 고객 관리용으로 나중에.
  - 이유: 영업은 대면 미팅 기반. 노트북으로 PPT 보여주는 게 더 효과적.
- **자동 DM 발송 연동**: 수집 시스템은 이미 별도 존재. 지금은 수동 발송으로 충분.
  - 이유: 자동 발송은 스팸 리스크. 초기에는 수동이 안전.
- **AI 기반 코멘트 생성 (GPT 연동)**: 조건문 기반 코멘트가 먼저.
  - 이유: GPT 코멘트는 품질 편차 큼. 검증된 템플릿 문구가 영업에 더 안정적.
- **실시간 모니터링/알림**: 계약 후 제공할 부가 서비스.
  - 이유: MVP 단계에서는 1회성 진단으로 충분.
- **타겟 분석 차트 (성별/연령/요일)**: 네이버 데이터랩 크롤링 필요. 복잡도 높음.
  - 이유: 있으면 좋지만 없어도 핵심 설득(점수+경쟁사 비교)은 가능.
- **저장 수 크롤링**: 네이버가 비공개 처리하는 경우 있음. 크롤링 난이도 높음.
  - 이유: 확보 가능하면 Phase 1.5로 빠르게 추가. 불가능하면 스킵.

## 성공 기준 (KPI)

### 시스템 효율 지표
- **업체당 제안서 생성 시간**: 목표 3분 이내 (현재 수동 30분~1시간)
- **일일 처리량**: 목표 50개 이상 (현재 수동 5~8개)
- **크롤링 성공률**: 95% 이상 (실패 시 fallback 작동)

### 영업 성과 지표 (30일 목표)
- **미팅 전환율**: 콜드DM 발송 대비 미팅 성사율 5% 이상
  - 100건 DM → 5건 미팅
- **계약 전환율**: 미팅 대비 계약률 30% 이상
  - 5건 미팅 → 1.5건 계약
- **월 계약 건수**: 최소 3건 (평균 단가 86만원 기준 월 258만원)
- **핵심 측정 방법**: 엑셀에 [DM 발송일 / 미팅일 / 계약 여부 / 패키지] 컬럼 추가하여 추적

### 제안서 품질 지표
- **사장님 반응**: "이런 것도 알 수 있어요?" 류의 반응이 나오는가
- **경쟁사 비교 언급**: 미팅 중 사장님이 경쟁사 얘기를 먼저 꺼내는가 (꺼내면 성공)

## 배포 블로커 (지금 영업 나가는 데 빠진 것)

### CRITICAL -- 이거 없으면 영업 못 나감

1. **PPT 출력 품질 문제**
   - 현상: URL이 인코딩된 채로 노출, 키워드/순위/검색량이 0으로 표시
   - 원인: 크롤링 데이터가 PPT 매핑 시 제대로 전달되지 않는 케이스 존재
   - 해결: PPT 생성 전 데이터 검증 단계 추가, 빈 값 fallback 처리

2. **경쟁사 비교 기능 부재**
   - 현상: 현재 자기 업체 데이터만 수집. 비교 대상 없음
   - 영향: 리서치 결과 계약 전환율 1위 요소가 "경쟁사 리뷰 격차". 이게 없으면 설득력 반감
   - 해결: 동일 키워드 상위 5개 업체 크롤링 추가

3. **점수 체계 부재**
   - 현상: 개별 항목의 유무만 체크. 종합 점수 없음
   - 영향: "38점"이라는 한 마디가 10분 설명보다 강력. 없으면 임팩트 약함
   - 해결: 22개 항목 가중치 기반 100점 환산 로직

4. **API 키 하드코딩**
   - 현상: process.py 37~41번 줄에 네이버 API 키가 소스코드에 직접 박혀 있음
   - 영향: Git에 올라가면 키 유출. 보안 문제
   - 해결: .env 파일로 즉시 이동

### HIGH -- 있으면 계약률 올라감

5. **배치 처리 미지원**
   - 현상: 엑셀 첫 번째 행만 처리
   - 영향: 100개 업체 대상 영업하려면 100번 실행해야 함
   - 해결: 전체 행 순회 + 진행률 표시

6. **손실 환산 로직 부재**
   - 현상: 순위와 검색량은 보여주지만 "잃고 있는 고객 수"로 환산하지 않음
   - 영향: 사장님이 "그래서 뭐?"라고 느낄 수 있음
   - 해결: 순위별 클릭률 테이블 기반 추정 고객 수 계산

## CTO에게 요청

1. **크롤링 안정성 확인**: Playwright로 상위 5개 업체를 연속 크롤링할 때 네이버 차단(captcha, IP ban) 리스크가 어느 정도인지. 딜레이/프록시 필요 여부.

2. **네이버 검색광고 API 상태 확인**: 현재 코드의 `get_keyword_stats`가 실제로 데이터를 반환하는지. PPT 스크린샷에서 "PC: 0회, 모바일: 0회"로 찍힌 걸 보면 API 호출이 실패 중일 가능성.

3. **PPT 템플릿 구조**: python-pptx로 경쟁사 비교 표(가변 행)와 점수 게이지를 구현할 수 있는지. 아니면 표 형태로 대체해야 하는지.

4. **배치 처리 시 메모리/속도**: Playwright 브라우저를 업체마다 새로 여는 것 vs 하나의 브라우저에서 탭으로 처리하는 것. 100개 업체 기준 예상 소요 시간.

5. **저장 수 크롤링 가능 여부**: 네이버 플레이스에서 "저장" 수를 가져올 수 있는 방법이 있는지 기술 조사 필요. 가능하면 Phase 1에 포함할 가치가 충분함 (사장님이 모르는 지표 = 임팩트 큼).
