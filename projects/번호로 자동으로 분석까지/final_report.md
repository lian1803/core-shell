# 최종 리포트 — 번호로 자동으로 분석까지 (네이버 플레이스 자동 진단 시스템)

## 전체 평가: [조건부 준비] (Conditional Ready)
**사유**: 단일 업체(1건) 대상의 제안서 생성은 즉시 영업에 활용 가능할 수준으로 동작함. 그러나 기획 파트 간(CPO vs CDO vs PM) 슬라이드 수 합의가 완전히 어긋나 있으며, 엑셀 100개 대량 배치 처리 시 네이버 IP 차단 방어 로직이 검증되지 않아 대규모 자동화 배포는 위험함.

## Wave별 산출물 요약

| 단계 | 산출물 핵심 내용 | 검증자 평가 |
|---|---|---|
| Wave 1 | CPO(8슬라이드, 전환율 집중) / CDO(11슬라이드, 시각화 집중) / CTO(기술 제약 분석) | 비즈니스 목적은 뚜렷하나 CPO와 CDO의 방향성이 충돌함. |
| Wave 2 | CPO-CTO 합의(8장) / CTO-CDO 합의(11장) / PM 계획(11장) | **의사결정 붕괴**. 서로 다른 슬라이드 수로 합의하고 PM이 11장으로 확정함. |
| QA (Wave 4) | CTO 코드 리뷰 (95점 버그 수정) / QA 테스트 리포트 | **치명적 모순 발견**. PM은 11장으로 지시했는데 QA는 8장 템플릿으로 테스트하고 PASS 처리함. |

## 발견된 엣지케이스 + 보완 필요

| 분류 | 엣지케이스 상황 | 예상되는 문제 | 보완 필요 사항 |
|---|---|---|---|
| **CRITICAL** | **슬라이드 인덱스 매핑 오류** | 코드는 11장(PM지시)을 기대하나 실제 템플릿이 8장일 경우, 특정 슬라이드(예: 체크리스트)가 누락되거나 데이터가 엉뚱한 곳에 덮어씌워짐. (QA가 `_has_slide`로 에러를 회피했다고 하나, 이는 데이터가 버려졌다는 뜻임) | 실제 영업용 템플릿(8장인지 11장인지)을 하나로 확정하고 코드의 `SLIDE_INDEX`를 하드코딩해야 함. |
| **CRITICAL** | **배치 처리 중 네이버 캡챠(Captcha) 발동** | 50개 업체를 배치로 돌리면 총 300번(50*6)의 크롤링 발생. 딜레이가 있어도 동일 IP면 중간에 네이버가 차단함. | 대량 배치(10개 이상) 실행 금지. 필요 시 프록시(Proxy) 로테이션 도입 필요. |
| **HIGH** | **업체명에 특수문자 포함** | 업체명이 `A&B <미용실>` 인 경우, 파일명 저장 시 OS 오류가 발생하거나 python-pptx 내부 XML이 깨질 수 있음. | `ppt_generator.py` 파일 저장부 및 치환부에 특수문자 이스케이프/제거 정규식 추가. (코드 미확인) |
| **MEDIUM** | **경쟁사가 5개 미만인 니치 키워드** | 경쟁사가 2개밖에 없는 동네/업종의 경우, 평균 점수 계산 시 모수가 적어 왜곡되거나 UI 차트가 깨질 수 있음. | 경쟁사 0~4개일 때의 UI 차트 예외 처리 확인 필요. (코드 미확인) |
| **MEDIUM** | **업종별 불필요한 평가 항목 (예: 메뉴/가격)** | B2B 업체나 서비스업(예: 변호사, 청소업체)은 '메뉴/가격' 항목이 없을 수 있는데 감점됨. | 업종(카테고리)별 평가 항목 제외/가중치 조정 로직 필요. |

## 비즈니스 로직 검증

| 비즈니스 요구사항 | 반영 여부 | 검증 의견 |
|---|---|---|
| **1. 사장님에게 '충격'을 주는가?** | 반영됨 | 경쟁사 비교 차트와 95점 만점 기준의 D등급 등 시각적 충격 요소는 구현됨. |
| **2. 손실을 '숫자'로 보여주는가?** | **미반영 의심** | CPO의 핵심 기획이었던 "현재 순위 클릭률 = 잃고 있는 고객 수" 로직이 PM 계획과 QA 리포트에서 완전히 증발함. |
| **3. 처리 시간 '3분' 이내인가?** | 반영됨 | 1개 업체 처리 시간 약 2.5분으로 KPI(3분) 달성. |
| **4. 실패 시에도 제안서가 나오는가?** | 반영됨 | 경쟁사 데이터 수집 실패 시 기본값(default)으로 처리되어 파이프라인 중단을 막음 (Graceful degradation 적용). |

## 독립 검증자 종합 의견

**"엔지니어링은 훌륭하나, 기획 소통이 무너진 프로젝트입니다."**

**👍 좋은 점:**
CTO의 기술적 대처가 매우 뛰어납니다. 외부 API 호출의 불안정성을 세마포어(동시성 제어), 랜덤 딜레이, 예외 처리로 훌륭하게 방어했습니다. API 키 보안 문제도 깔끔하게 해결했습니다.

**👎 나쁜 점:**
기획 파트의 소통 부재가 심각합니다. CPO는 8장을 합의했고 CDO는 11장을 합의했는데, PM은 11장으로 일정을 짜고, QA는 8장짜리 템플릿으로 테스트한 뒤 통과시켰습니다. 또한 CPO의 가장 핵심적인 비즈니스 로직(고객 손실 계산)이 실무 개발 단에서 조용히 누락되었습니다.

**🔥 가장 우선 해결할 것 3가지:**
1. 8장 vs 11장 PPT 템플릿 단일화 및 코드 동기화 (`config.py`의 `SLIDE_INDEX` 수정)
2. CPO가 기획한 '키워드 검색량 기반 예상 손실 고객 수' 계산 로직 복구
3. 네이버 API 키가 과거 Git 커밋 내역에 남아있을 가능성이 높으므로, 네이버 개발자 센터에서 기존 키를 즉시 삭제하고 **새 키로 재발급**
