## 합의 1: MVP 범위 — 무엇을 넣고 무엇을 뺄 것인가

### CPO 원안 (5개 기능)
1. 기본 크롤링 안정화
2. 22개 항목 + 100점 점수 계산
3. 경쟁사 비교 크롤링
4. PPT 템플릿 재설계 (심리 흐름 반영)
5. 엑셀 배치 처리

### CTO 검토 결과
- 전체 5개 기능 모두 구현 가능
- 총 예상 구현 시간: 약 3시간
- 단, 모듈 분리(코드 재설계)를 선행 작업으로 추가 요청 (30분 추가)

### 합의 결과: CPO 원안 전체 채택 + CTO 선행 작업 1개 추가

MVP Phase 1 최종 범위 (구현 순서):

| 순서 | 작업 | 담당 | 예상 시간 |
|------|------|------|----------|
| 0 | config.py 분리 + .env 생성 + 모듈 구조 재설계 | BE | 45분 |

- 이유: 핵심 영업 가치(점수 + 경쟁사 비교)와 무관. 영업 투입 후 피드백 받고 결정.

## 합의 2: 경쟁사 크롤링 — 비즈니스 임팩트 vs 기술 리스크 절충

### CPO 관점 (비즈니스 임팩트)
- 경쟁사 비교는 계약 전환율 1위 요소
- "38점 vs 경쟁사 평균 72점"이 없으면 설득력 반감
- 반드시 MVP에 포함해야 함. 절충 없음.

### CTO 관점 (기술 리스크)
- 네이버 크롤링 차단(IP 밴) 리스크: HIGH 등급
- 해결책: 세마포어 3개 동시 제한 + 랜덤 딜레이(1~3초) + User-Agent 로테이션
- 예상 소요 시간: 업체당 30초 내 완료 (6개 업체 동시 크롤링)
- 차단 시 대응: headless=False 전환으로 우회 테스트 가능

### 합의 결과: 경쟁사 크롤링 포함. 단, 아래 3가지 기술 조건 준수.

1. **동시 실행 수**: asyncio.Semaphore(3) — 6개 동시 금지
2. **딜레이**: 업체 간 random.uniform(1.0, 3.0)초 적용
3. **실패 허용**: 경쟁사 5개 중 3개 이상 성공 시 진행. 0개 성공 시에만 경쟁사 비교 슬라이드를 "수집 불가"로 표시

추가 합의 — 경쟁사 범위 정의:
- CPO 질문에 대한 CTO 제안 수용: "메인 키워드 검색 결과 상위 5개"
- 근거: 사장님에게 보여줄 때 "이 키워드로 검색하면 상위에 이 업체들이 나옵니다"라는 설명이 직관적. 동네+업종 필터링은 복잡도 대비 임팩트 미미.

## 합의 3: PPT 슬라이드 수 — 8슬라이드 vs CDO 제안 11슬라이드

### CDO 제안 내용 (CTO가 전달)
- 현재 8슬라이드 → 11슬라이드로 확장
- 추가 요청: 경쟁사 비교 슬라이드 1장 + 점수 게이지/차트 슬라이드 1장 (= +2장)
- 합산 시 10장. CDO가 별도 1장 더 추가 제안한 것으로 보임.

### CPO 관점
- 현재 CPO 설계: 8슬라이드 (표지~CTA 5단계 감정 곡선)
- 경쟁사 비교 슬라이드는 이미 슬라이드 4번에 통합 설계됨
- 점수 게이지는 슬라이드 3번에 통합 가능

즉, CDO 요청의 핵심 내용(경쟁사 비교 + 점수 시각화)은 CPO 8슬라이드 구조 안에 이미 포함됨.

### CTO 관점
- 슬라이드 수 증가는 python-pptx 템플릿 인덱스 관리 복잡도 증가
- 슬라이드 추가/순서 변경 시 SLIDE_INDEX 상수 업데이트 필요
- 11슬라이드가 불가능하지는 않으나 테스트 부하 증가

### 합의 결과: 8슬라이드 유지. 단, 콘텐츠 밀도 높임.

| 슬라이드 | 내용 | CDO 요청 반영 여부 |
|---------|------|-------------------|
| 1 | 표지 + 신뢰 확보 | - |
| 2 | 현재 상태 요약 (인정) | - |
| 3 | 종합 점수 100점 + 게이지 시각화 | 점수 게이지 통합 |
| 4 | 경쟁사 비교 표 (우리 vs 상위 5개) | 경쟁사 비교 통합 |
| 5 | 손실 환산 (잃고 있는 고객 수) | - |
| 6 | 키워드 전략 | - |
| 7 | 솔루션 요약 (패키지 3종) | - |
| 8 | CTA (행동 유도) | - |

슬라이드 11개로 가는 경우: 사장님 미팅 현장에서 "슬라이드가 너무 많다"는 피드백이 3회 이상 발생 시 재검토. 지금은 8슬라이드가 최적.

## 합의 4: API 키 보안 이슈 — CPO 관점 우선순위

### 현황
- process.py 37~41번 줄에 네이버 API 키 하드코딩
- CTO가 CRITICAL 등급으로 분류
- Git에 이미 올라갔을 가능성 있음

### CPO 관점
비즈니스 관점에서도 CRITICAL로 동의. 이유 3가지:

1. **영업 신뢰도 리스크**: API 키 유출 → 네이버가 API 사용 차단 → 시스템 전체 중단 → 영업 현장에서 제안서 못 뽑는 상황 발생
2. **팀 확장 시 리스크**: 영업 담당자가 코드를 받을 때 키가 같이 전달됨. 퇴사자 발생 시 키 로테이션이 번거로워짐
3. **외주 개발 시 리스크**: 나중에 코드를 외부에 보여주거나 공유할 때 키 유출

### 합의 결과: 모든 작업보다 API 키 .env 이동이 먼저 (구현 순서 0번으로 고정)

구체적 처리:
- config.py에서 python-dotenv로 환경변수 로드
- .gitignore에 .env 추가 (없으면 생성)
- 이미 Git에 올라간 키는 CTO가 네이버 콘솔에서 즉시 재발급 조치
- .env.example 파일 생성 (키 없이 변수명만)

```
[즉시, 15분] config.py + .env + .gitignore — API 키 보안
  ↓
[1일차 오전, 30분] 모듈 구조 분리 (crawlers/, api/, scoring/, generators/, io/)
  ↓
[1일차 오후, 30분] 22개 항목 + 100점 점수 엔진
  ↓
[2일차 오전, 45분] 경쟁사 크롤링 (세마포어 3, 랜덤 딜레이)
  ↓
[2일차 오후, 30분] 저장 수 / 새소식 / 답글 응답률 신규 항목
  ↓
[3일차 오전, 30분] PPT 재설계 (8슬라이드, 심리 흐름, 비교 표, 점수 게이지)
  ↓
[3일차 오후, 20분] 배치 처리 + 통합 테스트
  ↓
[4일차] 실전 영업 투입
```

## 미결 사항 (CDO/PM에게 전달)

1. **PPT 템플릿 디자인**: CDO가 8슬라이드 기준으로 슬라이드 3(점수 게이지)과 슬라이드 4(경쟁사 비교 표) 시각 디자인 제공 필요. python-pptx가 읽을 수 있는 .pptx 템플릿 형태로 전달.

2. **점수 항목 최종 확인**: CTO 설계의 22개 항목 중 카테고리 C-17번(키워드 순위)과 카테고리 E(검색 노출)가 중복. CPO 결정: 카테고리 E를 보너스 점수로 유지하고 17번을 삭제. 총점 100점 유지하되 C카테고리를 16점으로 조정.

3. **사장님 답글 응답률 표기**: CTO가 지적한 대로 근사치(최근 20개 리뷰 기준). PPT에 "최근 리뷰 기준 추정치"라는 작은 주석 추가. 이는 사실 왜곡이 아니라 투명성이며 신뢰도를 오히려 높임.

4. **저장 수 크롤링 실패 시 처리**: 업종별로 저장 수가 안 보이는 경우 있음. "확인 불가"로 표시하되 PPT에서 해당 항목 점수를 0점이 아닌 "측정 불가"로 처리. 경쟁사도 동일 조건이므로 비교에서 제외.
